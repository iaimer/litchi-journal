# 荔枝日记 Flutter 后续开发 Plan 文档

## 1. 当前版本定位

当前 Flutter 端已经进入"文字记录功能对齐版"。这意味着它已经基本满足日常文字记录需求，可以作为日常使用版本继续观察。

当前已完成的核心链路包括：

随手记、觉察、小确幸的添加、AI 润色、自动标签、编辑、删除；焦虑四问的新建、AI 润色当前回答、返回修改、草稿保护、编辑替换；右下角快速记录 FAB 扇形入口；习惯追踪的饮水、运动和 checkbox 更新；跨日自动创建新日记；AI 设置页；可编辑润色提示词和人生教练提示词；Obsidian 后台同步。

后续开发不应继续无序搬 Web 功能，而应按真实使用优先级推进：先稳定当前文字记录闭环，再规划图片与画廊，再迁移历史和统计，最后统一进行 Open Design UI 重设计。

---

## 2. 后续开发总原则

第一，Web 端是功能和数据行为参考，不是 Flutter UI 复刻目标。Flutter 应优先适配手机端交互。

第二，服务端仍然是 Markdown 写入的权威。Flutter 不直接修改 Obsidian 文件，不在本地拼完整日记模板，只发送清晰的操作意图。

第三，所有写入类能力都要考虑失败保护。保存失败不能丢输入，编辑失败不能关闭编辑面板，删除必须确认，跨日必须自动创建日记。

第四，AI 功能必须分场景处理。普通入口润色需要自动标签；焦虑四问润色不需要标签；人生教练属于回顾分析，不应混入普通输入润色链路。

第五，Open Design 应在功能稳定后统一进行，不要在每个功能开发中提前做大规模视觉重构。

## 3. P0：短期功能冻结与真实使用观察

### 目标

在继续新增大功能前，先用当前版本真实使用 1-2 天，观察是否还有阻碍日常记录的摩擦点。

### 需要重点观察

快速记录 FAB 是否能顺畅展开，五个入口是否不重叠、不误触。

随手记是否能通过统一记录页顺畅添加、润色、自动标签、编辑和删除。

觉察是否能通过统一记录页稳定自动标签，第一次写入时是否正确替换模板空 bullet，不产生多余 `- `。

小确幸是否能通过统一记录页正确写入、显示、编辑、删除。

焦虑四问是否能从 FAB 进入独立页面，新建、编辑、润色当前回答、保存后替换而不追加。

习惯追踪是否更新顺手，失败时是否能正确恢复状态。

草稿 2 分钟 TTL 是否合适。普通记录可能足够，焦虑四问如果经常被打断，可以考虑单独延长。

跨日自动创建日记是否稳定。新日期首次打开和首次写入都应正常。

编辑标签是否能写回 Obsidian。需要主动测试修改标签，而不是只修改正文。

### 完成标准

连续 1-2 天真实使用，没有出现阻塞级问题。文字记录核心链路可以稳定使用。

## 4. P1：图片添加能力分析与实现

### 背景

图片添加此前被后置。现在文字记录闭环基本完成，图片功能可以作为下一阶段的重要能力。图片尤其适合亲子、小确幸、生活记录和未来画廊页面。

### 第一阶段：只做分析

不要一开始写代码。先分析 Web 端和服务端当前图片能力：

图片是否已有上传 API；图片是否会压缩；图片保存路径在哪里；Markdown 图片引用格式是什么；Obsidian 附件目录如何组织；服务端是否会生成文件名；是否支持多图；是否有图片压缩设置；Web 端如何处理本地 Ollama 或 AI 无关场景。

### 第二阶段：最小实现

优先做"给一条记录添加一张图片"，而不是多图和画廊。

建议先支持随手记或小确幸。用户选择图片，Flutter 压缩或交给服务端处理，服务端保存附件，Markdown 中插入图片引用。保存后刷新今日页，图片可以显示或至少以附件链接显示。

### 风险点

Flutter 权限、Android 相册访问、图片压缩、上传失败重试、文件命名冲突、Obsidian 同步路径、Markdown 引用格式、图片预览性能。

### 完成标准

用户可以从手机选择一张图片，写入某条记录，并在 Obsidian 后台看到正确 Markdown 图片引用。

## 5. P1：画廊页面规划

### 背景

画廊页面是 Flutter 端的新能力，原 Web 端没有。这是移动端可以超越 Web 的方向。

画廊不应在图片链路前实现。它依赖日记中的图片引用、附件路径、日期和可能的标签信息。

### 规划方向

GalleryPage 可以集中展示日记中的图片。第一版可以按日期倒序展示图片缩略图，点击后打开大图或跳转到对应日记日期。后续可以按标签、记录类型、小确幸、亲子记录等筛选。

### 数据来源

第一阶段可以从 Markdown 图片引用中解析图片路径。未来可以建立媒体索引，缓存图片元数据，比如日期、来源 section、关联记录文本、标签。

### 不建议第一版做的内容

不做复杂瀑布流，不做 AI 图像识别，不做相册级管理，不做图片编辑，不做云图库。

### 完成标准

App 中新增画廊入口，可以浏览已写入日记的图片，并能关联到日期或记录。

## 6. P2：历史页 / 过往日记

### 背景

当前 Flutter 主要是今日页。Web 端具备过往日记查看能力。移动端未来也需要回看历史。

### 第一版范围

只读历史页。支持选择日期，读取对应日记，使用现有 DiaryMarkdownView 渲染。先不做历史条目的编辑删除。

### 后续扩展

历史页稳定后，再考虑历史记录编辑、删除、AI 回顾、按标签过滤、关键词搜索。

### 风险点

历史日期不存在时如何显示；是否自动创建历史日记；历史页不应误触写入；编辑历史日记时 date 必须严格使用目标日期。

### 完成标准

用户可以在 Flutter 中查看过去某一天的日记内容。

## 7. P2：统计页

### 背景

Web 端已有统计功能。Flutter 后续可以迁移统计，但应排在历史页之后，因为统计依赖一段时间的数据积累。

### 第一版范围

优先统计习惯数据，比如饮水、步数、阅读、学语言、补充剂完成情况。

### 第二版范围

标签分布、记录类型频率、焦虑记录频率、小确幸数量、AI 润色使用情况等。

### 不建议第一版做的内容

不做复杂图表，不做深度分析，不做 AI 总结，不做导出。

### 完成标准

用户可以看到最近一段时间的习惯趋势和记录概览。

## 8. P2：人生教练功能

### 背景

Flutter 已经有 `coachPrompt` 配置，但人生教练功能尚未接入。

人生教练不同于普通润色。它不是输入框辅助，而是回顾分析类能力，需要读取日记内容，识别模式，给出反馈和建议。

### 需要先分析 Web 端

人生教练入口在哪里；读取哪些内容；prompt 如何组织；输出是否写入 Markdown；写入哪个 section；是否支持编辑或删除；是否可以重新生成；是否需要日期范围。

### 第一版建议

先做"当天人生教练反馈"。读取当天日记内容，调用 AI，根据 coachPrompt 生成反馈，写入人生教练 section。

### 风险点

用户隐私、AI 输出过度判断、是否保存到 Obsidian、重复生成如何处理、是否允许编辑删除。

### 完成标准

用户可以在 Flutter 中生成当天人生教练反馈，并写入 Obsidian 对应 section。

## 9. P3：标签管理设置

### 背景

Flutter 目前能读取 TagConfig，也能用 TagPicker 和 AI 自动标签。但还不能编辑标签体系。

### 是否需要迁移

短期不建议优先做。标签体系属于基础配置，一旦手机端支持修改，就必须考虑服务端同步、AI prompt、历史数据兼容、排序和删除后旧记录如何处理。

### 第一版建议

如果需要，可以先做"查看标签体系"，只读展示当前 domains、topics、methods。

### 第二版建议

再做增删改、排序、同步服务端、冲突处理。

### 完成标准

用户可以在 Flutter 中安全查看或管理标签体系，且不会破坏 AI 自动标签和历史数据。

## 10. P3：习惯管理设置

### 背景

当前 Flutter 支持更新五个默认习惯，但服务端 `/habit` 仍是硬编码字段：water、steps、reading、language、supplements。

### 迁移前提

如果要做自定义习惯，必须先改服务端数据模型和 API。不能只在 Flutter 端做 UI，否则会形成无法写入的伪功能。

### 第一版建议

暂缓。保持现有五个习惯更新即可。

### 后续方向

服务端改为动态习惯配置后，Flutter 再迁移习惯管理页面，支持新增、隐藏、排序、单位、目标值、快捷按钮配置。

## 11. P3：远程 API / 同步设置

### 背景

SetupScreen 当前负责首次服务器地址和 Token 配置。SettingsScreen 已包含连接信息和 AI 设置，但尚未完整支持修改远程 API 配置。

### 第一版建议

SettingsScreen 中显示当前服务器地址和 Token 已配置状态，继续保持只读。

### 第二版建议

支持修改服务器地址和 Token，但必须先测试连接，成功后才能保存。修改失败不能让 App 失联。

### 风险点

用户误改 baseUrl 或 Token 后无法恢复；需要重新加载日记和 TagConfig；已有缓存是否清理。

## 12. P4：Open Design UI 重设计

### 背景

项目原则已经明确：Flutter 不复刻 Web UI。当前 UI 只是功能可用层，后续应通过 Open Design 统一重做。

### 适合启动的时机

当文字记录闭环稳定、图片/画廊方向明确、历史页是否优先有判断之后，再启动 Open Design。否则视觉设计会被功能变化反复打断。

### 设计目标

今日页应围绕手机端高频动作组织：快速记录、习惯、今日内容流、焦虑四问、小确幸、AI 润色和标签选择。

条目卡片、输入区、操作菜单、习惯卡、焦虑四问流程、设置页、画廊入口都应统一设计。

### 不建议做的事

不在功能未稳定时重做全局 UI；不为了复刻 Web 而牺牲手机端体验；不一次性重构所有组件。

### 完成标准

形成一套适合 Flutter 原生体验的荔枝日记视觉和交互语言。

### 16.6 已知注意事项

- 真机验证前必须先构建对应模式，例如：

```bash
/Users/yezi/development/flutter/bin/flutter build apk --release
adb -s <device-id> install -r build/app/outputs/flutter-apk/app-release.apk
```

- 如果只构建 debug，却安装 release，会导致真机不是最新代码。
- 日常覆盖安装不要使用 `flutter install --release`，因为它可能先卸载旧版并清空本地 baseUrl/token。
- 使用 `adb install -r` 覆盖安装在同 packageId、同签名、非降级安装时可以保留本地配置和 token。
- 当前环境有时会出现 ADB daemon 无法绑定本机端口：`could not install *smartsocket* listener: Operation not permitted`。这属于本地执行环境限制，不代表 App 崩溃。
- 如果 Codex 环境内 `adb install -r` 被 ADB daemon 权限限制挡住，可在本机终端执行相同覆盖安装命令后继续真机验收。
- 手机上 App 名称是「荔枝日记 dev」，applicationId 是 `com.example.litchi_journal_flutter`。

### 17.3 新增稳定性规则

后续修改必须保留以下规则：

- 启动配置读取必须有超时/异常兜底，不能让 App 无限 loading。
- 今日日记加载必须有超时/错误兜底，不能让今日页无限 loading。
- 空日记或缺少 habit section 时，今天页必须仍显示可交互习惯入口。
- `TagRepository.loadTagConfig()` 失败时返回 `DefaultTagConfig.value`。
- 标签缓存读写失败不能导致标签功能不可用。
- `HomeScreen._effectiveTagConfig` 在远程标签配置尚未加载完成时，也应返回默认标签配置。
- SettingsPage 的「标签设置」入口不能静默失败；远程标签失败时使用默认标签配置打开。
- 习惯图标渲染统一走 `HabitIcon`，不要直接 `Text(icon)`。
- Flora 图标逻辑名和旧 emoji 用户配置需要并存兼容。
- 品牌资源更新必须从 `docs/design-reference/` 源图派生，不要重新绘制近似图。

### 17.5 当前真正推荐的下一步

短期仍建议继续「稳定观察 + 小修」，不要立刻做大功能。

建议优先级：

1. 继续观察 SVG 图标在深色/浅色模式下的可读性。
2. 检查设置子页是否还有类似“静默失败”的入口。
3. 整理真实使用中遇到的阻塞 bug，优先修复而不是新增页面。
4. 功能稳定后，再规划完整日历历史页、画廊页或 Open Design 全局 UI 重设计。
